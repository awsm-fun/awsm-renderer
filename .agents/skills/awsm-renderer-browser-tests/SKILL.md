# Running the browser-only tests

CI (`.github/workflows/test.yml` = `task lint` + `cargo test --all-features`)
runs pure Rust logic on a host with **no GPU and no browser**. Anything needing
a real WebGPU device, canvas, shaders, or the Basis worker can only be proven in
a browser and is therefore **not in CI**. This skill runs that tier.

## What this runs

**`examples/player-tests` (the automated tier — this is the target).** A wasm
harness that loads every baked bundle in `examples/test-scenes/<scene>/bundle`
through the *real* player loader on a live GPU, then prints to the browser
console:

```
PLAYER-TEST <name>: PASS — <detail>
PLAYER-TEST <name>: FAIL — <detail>
...
PLAYER-TESTS COMPLETE: <pass>/<total>
```

It auto-runs on page load and **always** emits a `PLAYER-TESTS COMPLETE` line —
even on a panic (`PLAYER-TESTS COMPLETE: aborted (panic)`) — so waiting on that
line never hangs.

Not covered here (no machine-readable pass/fail — need human/vision judgment,
run only if explicitly asked): the **model-tests** viewer (`task model-tests`,
:9080) and the **golden screenshots** in `examples/test-scenes/*/golden.png`
(visual references, regenerated by replaying a scene's `author.js` through
`task mcp-dev` + the editor MCP). See the bottom section.

## Prerequisites

- `task` (go-task) on PATH — this repo drives everything through it.
- A browser-automation MCP with WebGPU: `chrome-devtools` (preferred — used
  below) or `claude-in-chrome`. If its tools are deferred, load them with
  ToolSearch first.
- Ports (from `taskfiles/config.yml`): **test-scenes = 9084**,
  **player-tests harness = 9091**. Do not guess these — they have changed before.

## Procedure

1. **Start both servers** (the harness fetches bundles from the test-scenes
   server — `task player-tests` does *not* start it for you). Run each with
   Bash `run_in_background: true`:

   ```
   task test-scenes     # http-server on :9084 (bundles)
   task player-tests    # trunk serve on :9091 (builds + serves the wasm harness)
   ```

   If a port is already serving (`curl -s -o /dev/null -w '%{http_code}' http://127.0.0.1:9084/`),
   don't start a second one.

2. **Wait for the harness to build.** `task player-tests` runs `trunk serve`,
   which compiles the wasm — the first build takes a while. Poll until
   `http://127.0.0.1:9091/` returns 200 AND a `*_bg.wasm` is being served (a
   plain 200 on `/` can precede the wasm being ready). Do not open the page
   before the build settles or the harness loads stale/empty.

3. **Open a fresh tab** to `http://localhost:9091` via the browser MCP
   (`new_page` / `tabs_create_mcp`). Never reuse another session's tab. Reload
   with cache disabled if the page was opened before the servers were ready.

4. **Wait for completion.** Poll the console until a line contains
   `PLAYER-TESTS COMPLETE:`. Give it up to ~2 minutes (many scenes, real GPU
   uploads). If it never appears, treat as failure (see Troubleshooting).

5. **Read the result — do NOT dump the whole console.** It is huge and will
   overflow context. `list_console_messages` saves its output to a file when it
   is large; **grep that file** (or filter by pattern if the tool supports it)
   for:
   ```
   PLAYER-TEST |PLAYER-TESTS COMPLETE
   ```
   Collect every `PLAYER-TEST` line plus the final `COMPLETE` line.

6. **Report** to the user: the `<pass>/<total>` from the COMPLETE line, and the
   full `<name>: FAIL — <detail>` text of every failure (quote them verbatim).
   State PASS only when `<pass> == <total>` and no `FAIL` / `aborted` line
   appears. Note that the background server processes are still running (offer
   to stop them, or leave them for re-runs).

## Interpreting / troubleshooting

- **Any `FAIL`, `aborted`, or missing `COMPLETE`** = the run failed; surface it,
  don't paper over it.
- **`COMPLETE: aborted (panic)`** — a check panicked; the preceding console
  error line has the payload. Report it.
- **Blank screenshots / readbacks reading 0 / no COMPLETE after 2+ min** — the
  Chrome tab may be *frozen*, not the renderer. Restart the browser and re-open
  the tab before blaming the code.
- **Stale results after a rebuild** — trunk's live-reload watcher is
  **mtime-blind**: it rebuilds on real content changes, not `touch`. If you
  edited renderer code and the harness looks stale, hard-reload the tab (ignore
  cache); if trunk didn't rebuild at all, a real one-line content edit to a
  watched crate forces it.
- This tier needs a **WebGPU-capable** browser; a software/no-GPU browser will
  fail uploads. If every scene fails at device/upload, check the browser, not
  the bundles.

## Optional: the visual tiers (only if explicitly asked)

These have no automated pass/fail — you must look.

- **model-tests viewer:** `task model-tests` (:9080, plus its media servers via
  `task model-tests-dev`). Open a model, use the built-in MSAA/SMAA toggles,
  compare by eye.
- **Golden screenshots:** `examples/test-scenes/<scene>/golden.png`. To
  re-verify or regenerate, bring up `task mcp-dev` (editor :9085 + MCP), replay
  the scene's `author.js` via `editor_dispatch_json`, then `save_project` /
  `export_player_bundle` / `screenshot_scene`. Goldens are window-dependent
  visual references, deliberately **not** byte-exact CI locks — compare
  structurally.
