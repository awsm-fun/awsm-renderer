# Dev disk usage — the `target/` directory

`target/` grows without bound under normal development (it has been observed
at **149 GB**, and a full disk corrupts builds in confusing ways — `task lint`
failing with "No space left on device" mid-compile, query-cache write errors,
and editor-tab instability).

## Safe cleanups, cheapest first

| What | Command | Cost |
|---|---|---|
| Incremental caches (biggest single win — tens of GB) | `rm -rf target/debug/incremental` | next build of each crate recompiles from scratch once |
| Old wasm artifacts | `rm -rf target/wasm32-unknown-unknown/debug/incremental` | same, wasm side |
| Everything (nuclear) | `cargo clean` | full workspace rebuild (~minutes native + wasm) |

`cargo clean -p <crate>` exists for single-crate surgical cleans but rarely
frees much — the incremental caches dominate.

## When to check

- Before long autonomous batches: `df -h /` — if available space is < 20 GB,
  clean the incremental caches first and say so in the session log.
- The day-3 loop's standing gate: every heavy build is preceded by a disk
  check; a low-disk clean is REPORTED, never silent.

## What NOT to delete

- `target/` while `trunk serve` or `task mcp-dev` is running (the watcher
  holds artifact locks; killing files under it wedges the build — restart the
  stack after a clean).
- Anything outside `target/` — `media/` holds test assets, `dist/` is
  regenerated by trunk but deleting it mid-serve breaks the dev server.
