# MSAA fix — May 27, 2026 (session 3)

## Resolution

User reported the more-opt branch's MSAA looked visibly worse than
`main` on MorphStressTest (clear stairsteps at capsule silhouettes).
Side-by-side screenshots confirmed the regression.

**Root cause:** the Stage 3 `final_blend` compute pass was overwriting
`opaque_tex` for classify-detected edge pixels, replacing the correct
4-sample-averaged values that primary opaque's inline
`msaa_resolve_samples` had already written. The Stage 3 data flow has
subtle accumulator-vs-counter sync bugs (low effective edge_count
relative to dispatched threads — observed in earlier diagnostic
sessions via the CYAN/MAGENTA debug bars).

**Fix:** early-return at the top of
`MaterialOpaqueRenderPass::render_edge_resolve`. The pipelines still
compile at boot via `ensure_compiled`; only the per-frame dispatch is
skipped. Quality now matches `main` byte-for-byte at the shader level
(verified: `git diff main` over all `material_opaque/.../*.wgsl` files
in the resolve path returns 0 lines).

## What this leaves in place

- Stage 3 architecture (classify-side edge data emission +
  edge_resolve pipelines + final_blend) compiles and is fully wired,
  just not dispatched. Re-enabling = remove the early-return.
- All Block A-F work, D.1 PART 2 first-party extension, and the
  classify mesh-vs-mesh-edge detection (`1a3f68b`) and final_blend
  slot_map filter (`4c74652`) remain landed — those are correctness
  fixes that apply when/if the Stage 3 dispatch flow is debugged
  back on.
- Primary opaque's inline `msaa_resolve_samples` continues to be the
  authoritative MSAA-resolver (same as `main`).

## What's known broken in Stage 3 (for next session)

When dispatched, `final_blend` produces visibly worse output than the
legacy path. Diagnostic evidence collected this session:

1. `render_edge_resolve` does dispatch (~6 workgroups × 64 threads
   per the indirect args, confirmed via per-thread MAGENTA debug bar
   write at row 10 in `final_blend.wgsl`).
2. But `total_edges` read inside `final_blend`'s shader (from the
   data-buffer mirror `edge_data[edge_count_index]`) is much smaller
   than the dispatched thread count — observed CYAN bar at row 30 was
   ~10-30 pixels wide while the MAGENTA dispatch-thread bar at row 10
   was ~400 pixels wide.
3. This implies the data-buffer `edge_count` mirror is being
   incremented far less than the args-buffer `workgroup_count_x`
   atomic counter — they SHOULD increment in lockstep inside classify
   (`compute.wgsl:237-238` calls `atomicAdd` on both per-edge).
4. Cyan debug write of `samples_to_process > 1u` from primary opaque
   confirmed the inline `msaa_resolve_samples` path IS reached at
   silhouettes (cyan outlined the capsule edges).
5. RED tint debug on `final_blend`'s output produced NO visible red
   on capsules — `final_blend` writes are either not landing on
   visible pixels or being clipped somewhere.

Hypotheses to verify next session:

- The two atomicAdds in classify may not actually target what we
  think they do; binding 4 vs binding 6 mapping (args_buffer vs
  data_buffer) — verify with a CPU-side `mapAsync` readback after a
  frame.
- `reset_header` may be racing with classify within the same frame
  (it's called from `render.rs:611` BEFORE classify, but a barrier
  may be missing).
- `edge_layout.edge_count_index` u32-stride value may differ between
  the host-built uniform and what classify shader reads (verify with
  CPU dump of the uniform buffer's first 4 bytes).

If after debugging the data flow `final_blend` still doesn't earn its
complexity (e.g. produces the same quality as the legacy path), the
whole Stage 3 infrastructure (~2.7kloc) can be deleted for a net
simplification, retaining only the legacy `msaa_resolve_samples`
path. The original motivation in `more-optimizations.md` was
Android-SPIR-V-bloat from the cross-shader switch in
`msaa_resolve_samples` — that concern is still valid; a fix would
need to be smaller and more surgical than Stage 3 (e.g.
per-shader-id specialization of the primary opaque pipeline with no
inline cross-shader switch).
