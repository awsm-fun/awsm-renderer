## What the user sees

Screenshot shared shows 8 colored capsules (yellow/green/blue-tall/magenta/orange/lime/purple/cyan) sitting on a white pedestal, against a 3D-room photo-studio environment.

**Visible aliasing**: Right edges of capsules show distinct stairstep pattern. The left edges look smoother (asymmetric). The pattern says: edge_resolve isn't blending properly on certain edges.

## What's known good

- `MSAA ON pre-fix-4c74652`: hard color jump at edges (samples showed 1-pixel transition magenta→dark)
- `MSAA ON post-fix-4c74652`: ~4-pixel intermediate-color gradient at magenta edge (better, but user says still bad)
- Fox model: looks smooth — but Fox has low-contrast silhouettes so may mask any remaining bug
- Reverting D.1 PART 2 first-party extension (commit `e105569`) DOES NOT fix the aliasing — confirms it's pre-existing, NOT caused by my work

## Key code paths

- `final_blend.wgsl` — reads accumulator[4 slots], blends. **PRE-FIX READ ALL 4 SLOTS UNCONDITIONALLY**, picking up stale data from prior frames.
- `edge_resolve.wgsl` (per shader_id) — scans slot_map for own bucket_index, writes accumulator[edge_pixel_id*4+slot_index]
- `skybox_edge_resolve.wgsl` — scans slot_map for 0xFE (skybox sentinel), writes accumulator slot
- `classify/compute.wgsl` — emits edge data, packs slot_map = (seen[0..3]) where seen[] = distinct bucket_indices (or 0xFE skybox, 0xFF empty)

## Hypotheses to investigate

1. **MSAA-4 isn't actually MSAA-4** — geometry pass might write same data to all 4 samples (no real multi-sampling). Check geometry pipeline's multisample state.

2. **slot_map encoding mismatch** — classify writes `seen[i] & 0xFFu` where seen[i] can be `bucket_index` (0..N) or `0xFEu` or initial `0xFFu`. If bucket_index ever exceeds 253, byte truncation could produce 0xFE collision. Check.

3. **Sample-mask vs slot-map mismatch** — classify writes sample mask per shader_id and ALSO updates slot_map. If they disagree, edge_resolve picks wrong slot. Verify they're in sync.

4. **Per-sample shading produces near-identical colors** — for PBR with strong directional lighting, each sample within a pixel may produce nearly the same color (sample positions within a pixel are sub-pixel offsets, not enough to vary lighting). In which case edge_resolve "blending" doesn't visibly differ from sample-0.

5. **accumulator slot ordering bug** — final_blend reads accumulator[ep_id*4+slot_idx]; edge_resolve writes accumulator[ep_id*4+slot_idx]. If they use DIFFERENT slot_idx for the same shader_id (e.g. PBR writes slot 0 but final_blend tries to find PBR at slot 1), the data is misaligned.

6. **Floor (pedestal) is a separate PBR mesh** — capsule edges might be against the pedestal/floor (both PBR). seen_count=1 (only PBR), classify doesn't emit edge → no MSAA blending at THESE edges. Look at where capsules meet pedestal.
