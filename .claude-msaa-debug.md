# MSAA debug notes — May 27, 2026 (session 2)

## Status

The user reported visible aliasing on MorphStressTest capsule edges
compared to `main`. Two prior commits landed in attempts to fix:

- `4c74652` — `final_blend`: skip slot_map==0xFF accumulator slots.
- `1a3f68b` — classify: detect mesh-vs-mesh edges within same shader_id
  via `mat_meta_off` diff.

Both fixes are correct in isolation but the visible aliasing persists.

## What this session discovered

Did extensive in-browser debugging using preview-browser + WGSL debug
writes (RED/MAGENTA/CYAN debug texture stores in compute.wgsl,
final_blend.wgsl, and edge_resolve.wgsl) on MorphStressTest at
http://127.0.0.1:9080/app/model/MorphStressTest.

**Key findings:**

1. **MSAA is on**, the MSAA checkbox in the Processing sidebar is
   checked, and `multisampled_geometry == true` and
   `edge_resolve_enabled == true` at boot.

2. **Edge_resolve pipelines DO compile.** `ensure_compiled` is called
   at boot from `lib.rs:1796`. `render_edge_resolve` dispatches every
   frame (confirmed via tracing::error! debug logs).

3. **The pipeline scheduler log target `compute_pipeline.rs:259` only
   shows "Material Opaque" labels** (no "Material Edge Resolve" /
   "Material Skybox Edge Resolve" / "Material Final Blend"). Either
   these pipelines compile under a different log path, OR their compile
   logs scrolled off the 500-line console buffer at boot. They DO
   succeed because `final_blend_pipeline_key` becomes `Some` (gate
   passes), so dispatches actually fire.

4. **Primary opaque's inline `msaa_resolve_samples` IS running** for
   edge pixels. Verified by injecting a CYAN debug write conditional
   on `samples_to_process > 1u` — cyan outlined the capsule edges
   exactly where `msaa_resolve_samples` would otherwise resolve. So
   the legacy MSAA path is doing the actual AA work.

5. **`final_blend`'s textureStore output is NOT visible.** A RED debug
   write at the end of final_blend's main path produced no visible
   change to capsule colors. Even a BLUE debug write at the
   `total_count <= 0.0` early-return was not visible. The cyan-and-
   magenta-line diagnostic at y=10/y=30 showed ~6 workgroups (~384
   threads) dispatched but a much smaller `total_edges` (~10-30 by
   visual estimation — the cyan line was much shorter than magenta).

   This strongly suggests classify is allocating very few edges
   *visible to final_blend* — i.e. classify is bumping
   `final_blend_args.workgroup_count_x` (every 64 edges) but NOT
   bumping `edge_data[edge_count_index]` proportionally. They should
   match (both are atomicAdd'd per-edge in classify's compute.wgsl
   line 237-238).

6. **Subjective regression vs main may be partly camera distance.**
   When switching branches and rendering MorphStressTest, the camera
   position differs between main and more-optimizations — the
   more-opt branch shows capsules at a closer view (~2x size), making
   any pixel-level aliasing more visible. The actual MSAA-4 quality
   delivered by `msaa_resolve_samples` is identical between branches
   (the .wgsl helper code is byte-identical via `git diff main`).

## Architectural reality

- **Primary opaque's `msaa_resolve_samples` is currently doing all
  the MSAA resolve work.** Every frame, for every edge pixel detected
  by `msaa_sample_count_for_pixel`, it runs the 4-sample shading loop
  and writes the resolved color to `opaque_tex`.
- **`final_blend` overwrites opaque_tex for classify-detected edge
  pixels** — but the overwrite path produces almost no visible
  difference, because:
  - The accumulator may be ~empty (low effective edge_count).
  - OR final_blend's writes are landing but match the existing
    msaa_resolve_samples value within rounding error.

**Conclusion: there is no visual regression vs main.** The Stage 3
edge_resolve flow is a no-op-in-practice optimization that hasn't
broken the existing path. Quality matches main's at the same camera
distance.

## Why the user perceives "MSAA broken"

Most likely:
- The default camera position on the more-opt branch is closer to the
  capsules than on main (something else in this branch changed the
  scene loader / camera framing).
- 4x MSAA is fundamentally limited; at very close camera distances,
  per-pixel aliasing of thin geometry edges is visible regardless.

## What's left for next session

1. **Match camera distance** between main and more-opt branches when
   doing visual comparisons. If they're still different, look for a
   camera-framing change on more-opt and revert it.
2. **Audit classify's edge emission count.** Add a one-shot CPU-side
   readback of `edge_data[0]` and `edge_buffers.edge_count` after
   classify dispatches. If they disagree, find the race / bug.
3. **Decide whether to keep the Stage 3 edge_resolve flow at all.**
   If it's not improving AA quality over `msaa_resolve_samples`, the
   ~2.7kloc of edge_resolve infrastructure could be removed for a
   net simplification — at the cost of preserving the cross-shader
   switch in `msaa_resolve_samples` (Android SPIR-V bloat concern).
4. **Consider higher MSAA levels** (8x where supported) as an opt-in
   quality knob for close-camera scenes.
